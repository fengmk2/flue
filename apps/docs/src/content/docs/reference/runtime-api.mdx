The runtime package provides the primitives used by both workflow modules and message-driven agent modules. The signatures below are intentionally representative for layout evaluation during the docs frontend spike.

## `defineAgentProfile(...)`

Define reusable behavior that can be selected for detached delegation.

```ts
const researcher = defineAgentProfile({
	name: 'researcher',
	instructions: 'Research carefully and cite source files.',
});
```

| Parameter      | Type     | Description                                            |
| -------------- | -------- | ------------------------------------------------------ |
| `name`         | `string` | Stable name used to select the profile for delegation. |
| `instructions` | `string` | Behavioral guidance applied to the delegated agent.    |

### Usage guidance

Profiles are definitions, not instances or sessions. Attach a profile to a created agent before selecting it from `session.task(...)`.

## `createAgent(...)`

Create an initializer that supplies defaults for one runtime agent.

```ts
const agent = createAgent(() => ({
	model: 'anthropic/claude-sonnet-4-6',
	subagents: [researcher],
}));
```

| Option      | Required | Description                                               |
| ----------- | -------- | --------------------------------------------------------- |
| `model`     | Yes      | Provider/model identifier used for operations by default. |
| `subagents` | No       | Reusable profiles available to tasks.                     |
| `tools`     | No       | Tools available within agent operations.                  |

<Callout kind="tip" title="Provider identifiers">
	Model identifiers use the `provider/model-id` form. Provider credentials are supplied through
	environment variables or explicit provider configuration.
</Callout>

## `init(...)`

Initialize a created agent inside a workflow and return its harness.

```ts
export async function run({ init }: FlueContext) {
	const harness = await init(agent, { name: 'analysis' });
	const session = await harness.session();
	return session.prompt('Analyze the change set.');
}
```

| Parameter      | Type               | Description                                                       |
| -------------- | ------------------ | ----------------------------------------------------------------- |
| `agent`        | `CreatedAgent`     | The configured agent initializer.                                 |
| `options.name` | `string`           | Optional harness name for isolated workflow scopes.               |
| Return value   | `Promise<Harness>` | Initialized environment containing sessions and workspace access. |

### Harness naming

Omit the name for the default harness. Add a stable name when the workflow needs separate session histories or tool/sandbox contexts for more than one collaborator.

## `harness.session(...)`

Select a session in an initialized harness.

```ts
const defaultSession = await harness.session();
const reviewSession = await harness.session('review-thread');
```

| Argument | Default     | Description                               |
| -------- | ----------- | ----------------------------------------- |
| `name`   | `'default'` | Conversation boundary inside the harness. |

### Persistence boundary

A session holds message history and operation metadata. A harness may hold several sessions over the same agent workspace; an agent instance may outlive individual requests.

## `session.prompt(...)`

Run a prompting operation, optionally requiring structured data.

```ts
const response = await session.prompt('Classify the deployment risk.', {
	result: v.object({
		severity: v.picklist(['low', 'medium', 'high']),
		reason: v.string(),
	}),
});
```

| Option           | Type      | Description                                        |
| ---------------- | --------- | -------------------------------------------------- |
| `result`         | Schema    | Validates and types the operation response data.   |
| `response.data`  | `unknown` | Structured value when a result schema is supplied. |
| `response.usage` | `Usage`   | Token usage metadata from the provider.            |

<Callout kind="warning" title="Do not conflate sessions and runs">
	`session.prompt(...)` is an operation. It becomes part of a workflow run only when invoked inside
	a workflow; direct interactions with an agent instance remain session operations.
</Callout>

## `session.task(...)`

Delegate a focused unit of work with isolated message history while retaining access to the configured project context.

```ts
const research = await session.task('Find the relevant authorization code.', {
	agent: 'researcher',
	cwd: '/workspace/project',
});
```

### When to use tasks

Use tasks when research or analysis is naturally separable from the primary conversational flow. Use a second named session instead when a continuing discussion must retain its own history.
