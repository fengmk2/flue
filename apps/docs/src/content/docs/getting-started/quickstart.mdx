Flue gives an agent a programmable harness: model defaults, sessions, tools, and an optional sandbox. This walkthrough creates a small workflow that prompts an agent and returns a structured answer.

## Create a project

Start a project and install the runtime and CLI packages used by a Flue application.

```bash
mkdir hello-flue && cd hello-flue
pnpm init
pnpm add @flue/runtime valibot
pnpm add -D @flue/cli typescript
mkdir -p .flue/workflows
```

The `.flue/` directory is your application's authoring root. Workflows define orchestrated invocations; agents can be added separately when you need persistent message-driven behavior.

## Write a workflow

Create `.flue/workflows/hello.ts` with a created agent, an HTTP channel, and a `run()` entrypoint.

```ts
import { createAgent, http, type FlueContext } from '@flue/runtime';
import * as v from 'valibot';

export const channels = [http()];

const agent = createAgent(() => ({
	model: 'anthropic/claude-haiku-4-5',
}));

export async function run({ init, payload }: FlueContext) {
	const harness = await init(agent);
	const session = await harness.session();
	const response = await session.prompt(`Write a welcome sentence for ${payload.name}.`, {
		result: v.object({ message: v.string() }),
	});

	return response.data;
}
```

A workflow **run** begins when this exported function is invoked. The `harness` returned from `init()` owns the initialized agent environment, while `session` holds conversational history for operations such as `prompt()`.

## Run locally

Set a provider API key and invoke the workflow through the CLI.

```bash
export ANTHROPIC_API_KEY="your-api-key"
pnpm exec flue run hello --payload '{"name":"Ada"}'
```

| Flag        | Purpose                                     | Example            |
| ----------- | ------------------------------------------- | ------------------ |
| `hello`     | Selects the workflow module to invoke.      | `hello`            |
| `--payload` | Supplies JSON input to `run()`.             | `'{"name":"Ada"}'` |
| `--target`  | Selects a build/runtime target when needed. | `node`             |

<Callout kind="tip" title="Use a lightweight model while iterating">
	A smaller provider model keeps local design and workflow iterations responsive and inexpensive.
	Select the production model only when its behavior is required for testing.
</Callout>
